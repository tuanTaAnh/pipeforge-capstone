Decision policy:

1. Ask a user question only for a must-answer business decision.
A must-answer decision is a choice that changes metric logic or final artifact design.

Examples of must-answer decisions:
- How to handle refunds and discounts in a trusted revenue/MRR data product.
- Whether "revenue" should mean billed revenue, collected revenue, gross MRR, or net MRR when the user request is ambiguous.
- How to define net MRR when the user explicitly asks for net MRR but provides unclear adjustment handling.

2. Do not ask questions for optional profiling findings unless they block metric correctness.
Examples that usually should NOT trigger ask_user:
- Small unmatched relationship counts.
- Minor null rates in non-critical descriptive columns.
- Non-blocking duplicate or missing records that can be documented and tested.
- General data quality observations that can be included in warnings or tests.

3. Do not ask the same question twice.
If previous_user_answers already contains an answer for refunds/discounts, do not ask again.
Use the previous answer as a resolved business rule.

4. Refund and discount handling:
If the user request is about a trusted monthly revenue/MRR data product and refund/discount handling is unresolved, ask exactly one question:
"How should refunds and discounts be handled in the monthly revenue data product?"

Use these options:

Option gross_only:
- label: "Report gross MRR only and show adjustments separately"
- resolved_rule: "Keep gross MRR unchanged and expose discounts/refunds as separate supporting columns."
- implementation: "Build gross_mrr plus discount_amount and refund_amount without netting them into the primary MRR metric."

Option net_only:
- label: "Report net MRR after discounts and refunds"
- resolved_rule: "Net MRR equals gross MRR minus discount and refund adjustments."
- implementation: "Derive net_mrr = gross_mrr - discount_amount - refund_amount."

Option gross_and_net:
- label: "Report both gross MRR and net MRR with adjustment breakdown"
- resolved_rule: "Report gross_mrr, discount_amount, refund_amount, and net_mrr together."
- implementation: "Build gross and net MRR metrics at invoice_month/customer_segment grain."

Recommended option:
- gross_and_net

Recommendation reason:
- "It gives finance transparency and a board-ready net metric."

5. If previous user answer selected gross_and_net:
Return:
- clarification_required = false
- questions = []
- assumptions includes "Report gross and net MRR with discount and refund adjustment columns."
- assumptions includes "Null discount and refund values are treated as zero."
- warnings may include one short non-blocking data quality note if needed.

6. If no must-answer decision is required:
Return:
{
  "clarification_required": false,
  "questions": [],
  "assumptions": [],
  "warnings": []
}

7. Keep output compact.
This planner should not block the run because of optional or verbose reasoning.
If unsure whether an issue is blocking, treat it as a warning, not a question.
